MDN: Form Validation

source

  • Client-side verification: initial check done by the HTML/JS before submitting a form - check for things like proper formatting, all required fields, etc. NOT a great security measure
  • Form validation: why?
    • Get the right data in the right format
    • Protect users' data
    • Protect ourselves (the server/app/data)
  • Built-in form validation with HTML5:
    • required attribute
    • minlength/maxlength (strings)
    • min/max (numerical)
    • type
    • pattern
  • CSS pseudoclasses :valid and :invalid to verify by UI and allow/prevent form submission
  • Good UX + Accessibility guidelines: always indicate if a field is required, and only require info you actually need.
  • Regex = super useful, super complicated. More Regex Info
  • Use minlength/maxlength, but provide character count feedback (such as a visible character-count at the bottom of a field)
  • More validation-related attributes
  • JavaScript Constraint Validation (Constraint validation API)
    • set of methods/properties for various DOM elements (button, fieldset, input, output, select, textarea)
    • properties: allows validation with validationMessage, validity (various states), willValidate(boolean)
    • methods: checkValidity() (boolean), setCustomValidity() (custom error message)
  • Custom error messages -- can't change their look/feel with CSS, might display in a different language (browser locale dependent)
  • Use "novalidate" on the form to turn off browser-automated validation so the script can take over -- constraint API + pseudo-classes still work.
    • (use aria-live on error message span text to read the error message aloud when it shows up)
    • this also allows us to style the error messages with CSS
